# analyze リライト方針メモ

更新日: 2026-03-30
対象: `analyze` のスクリーニング出力と分類基準の見直し

## 目的

このメモは、2026-03-30 時点の最新 Probe / Survey 結果と、既存 `analyze.csv` の観察をもとに、`analyze` リライトの方針を整理したものです。

前提となる生データは以下です。

- `docs/new-environment-constraints.md`
- `docs/env-test-results/probe_Local_20260330_020742.txt`
- `docs/env-test-results/probe_OneDrive_Synced_20260330_020158.txt`
- `docs/env-test-results/survey_Local_20260330_020742.txt`
- `docs/env-test-results/survey_OneDrive_Synced_20260330_020158.txt`

個別マクロの `analyze.csv` は機密情報を含むため、この docs には含めません。

## 結論

`analyze` は「個別分析器」ではなく「スクリーニング器」として作り直します。

芯は以下です。

- 分類軸は `EDRリスク` `互換性リスク` `パスのハードコード` の3本に絞る
- `EDRリスク` は、実測でそのコードや構文自体が止められたものだけに限定する
- `相対パス` や `ThisWorkbook.Path` 前提は EDR ではなく、パス設計の問題として別軸に出す
- `SendKeys` や `AppActivate` のような不安定系は、実測遮断でない限り EDR に入れず、互換性または要レビューへ寄せる
- 推測を主表示に混ぜない。事実と観測を前に出す
- 出力は「全体一覧」と「ヒットコード一覧」を分ける
- 必要なら補助として「マクロ別詳細」を持つ

## 1. 新しい分類基準

### 1-1. EDRリスク

ここには、Probe で「そのコードや構文自体が止められた」と観測できたものだけを入れます。

2026-03-30 時点で少なくとも以下は EDR リスクとして扱います。

- `Declare PtrSafe Function` などの `Declare`
- VBA からの `WScript.Shell.Run`
- VBA からの PowerShell 起動

`Declare` は Local / OneDrive_Synced の両方で `BLOCKED` であり、保存後再オープン時にブック破損扱いになっています。これは最優先の実測根拠です。

### 1-2. 互換性リスク

ここには、古い依存物や不安定な GUI 自動操作など、「環境や構成で動作が崩れるもの」を入れます。

2026-03-30 時点で代表例は以下です。

- `DAO.DBEngine.36`
- `MSComDlg.CommonDialog`
- `MSCAL.Calendar`
- `DDEInitiate`
- `SendKeys`
- `AppActivate`
- `keybd_event`
- `Sleep`
- `InternetExplorer.Application` 依存

このうち `SendKeys` は empty 送信自体は Probe で通っているため、少なくとも現時点では EDR リスクではありません。

### 1-3. パスのハードコード

ここには、環境依存の保存先・参照先前提を入れます。

代表例は以下です。

- `C:\Users\...`
- `OneDrive - 会社名`
- 固定ドライブ文字列
- `ThisWorkbook.Path` を業務ルートとして使う前提
- 相対パス前提の保存・列挙

これはセキュリティ遮断ではなく、OneDrive / SharePoint / ローカル同期環境で壊れやすい設計の問題です。EDR リスクに混ぜません。

## 2. 事実・観測・推定を分ける

判定は最低でも以下の3段に分けます。

### fact

コード上に何が書いてあるか。

例:
- `Declare`
- `WScript.Shell`
- `DAO.DBEngine.36`
- `ThisWorkbook.Path`
- 固定パス文字列

### observed risk

Probe / Survey で実測された遮断や失敗に対応するもの。

例:
- `Declare` はブロックされる
- `WScript.Shell.Run` は失敗する
- `DAO.DBEngine.36` はクラス未登録で失敗する

### inference

コードから見て危なそうだが、まだ未実測のもの。

主CSVやツールチップは、原則として `fact` と `observed risk` を優先し、`inference` は補助メモにとどめます。

## 3. 出力の構成

### 3-1. 全体一覧

マクロ単位の主CSVを出します。

最低限ほしい列は以下です。

- `macro_name`
- `edr_risk`
- `compat_risk`
- `hardcoded_path_risk`
- `evidence`
- `review_note`

件数や代表的な根拠はここに載せますが、総量指標だけで濁らせません。

### 3-2. ヒットコード一覧

別CSVまたはJSONで、実際に引っかかったコード断片を一覧化します。

最低限ほしい項目は以下です。

- マクロ名
- カテゴリ
- 検出パターン
- モジュール名
- 行番号または行付近
- コード断片
- 根拠種別

これはレビュー担当者向けの根拠棚であり、推測ではなく実コードを置きます。

### 3-3. マクロ別詳細

必要なら 1マクロ1フォルダで詳細を残します。

候補は以下です。

- 元コード断片
- 検出結果 JSON / CSV
- 根拠メモ
- 最終判定

個別詳細は補助であり、主戦場はあくまでスクリーニング一覧とヒットコード一覧です。

## 4. 個別マクロごとのサブフォルダ

出力単位は、個別マクロごとにサブフォルダを切るほうがよいです。

理由は以下です。

- 後からレビューしやすい
- 判定と根拠を同居させられる
- 誤判定の洗い直しがしやすい
- スクリーニング結果から詳細へ自然に降りられる

## 5. ツールチップと表示文言の方針

ツールチップや主表示に無責任な分析を書きません。

書いてよいのは以下です。

- 何が検出されたか
- それが Probe / Survey 上どう観測されたか
- 何軸のリスクに置いたか

書かないほうがよいのは以下です。

- 未実測の断定
- EDR と互換性とパス問題の混同
- 「たぶん危険」だけで終わる曖昧な評価

## 6. 実装の優先順

リライトは以下の順で進めるのが自然です。

1. 分類基準を3軸へ整理する
2. EDR 判定を実測ベースへ狭める
3. パス問題を独立軸に切り出す
4. ヒットコード一覧を追加する
5. マクロ別サブフォルダ出力を追加する
6. ツールチップと表示文言を fact / observed risk 基準へ直す

## 7. 一文で言うと

`analyze` のリライト方針は、事実と観測を中心にした3軸スクリーニングへ戻し、全体一覧と根拠一覧を分離することです。
