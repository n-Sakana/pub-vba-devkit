# vba-devkit プロジェクトレビュー

日付: 2026-03-28
対象: 構想 / 設計 / 実装 / 運用
前提: 修正作業は行わず、現状レビューのみ

## 総評

このプロジェクトは、かなり筋が良いです。

良い理由は単純で、最初から「一般論として便利な VBA ツール」を作ろうとしていないからです。対象は明確に、64bit Office・EDR・OneDrive / SharePoint 移行という、実際に手を焼く条件へ絞られています。構想が狭く、狙いが鋭い。こういう道具は広げるより、刺さるほうが強いです。

しかも、その構想が実装段階で崩れていません。`Analyze` で静的に全体を洗い、`Probe` で新環境の現実を測り、最後に突き合わせる。README、`docs/overview.md`、`lib/Analyze.ps1`、`lib/Probe.ps1` がこの骨格で揃っています。構想だけ立派で実装が別物、というありがちな事故は今のところ起きていません。

端的に言えば、これは「単発スクリプトの寄せ集め」ではなく、ちゃんと移行実務の流れを持った道具箱になっています。紅茶で言えば、茶葉・湯温・抽出時間が別々に暴れていない段階です。

現状の判定はこうです。

- 構想: 強い
- 設計: 強い
- 実装: 実用水準を十分に超えている
- 運用: 一人運用の道具としてかなり成立している
- 保守性: 今後の伸びしろと負債が両方見えている

## 1. 構想レビュー

### 1-1. 問題設定が正しい

このプロジェクトの核は、「VBA を解析する」ことではなく、「新環境移行で壊れるものを、開かずに洗い出し、必要なものだけ直す」ことです。ここが非常に良いです。

`docs/overview.md` にある 3 層構成は、実務の順番として自然です。

1. `Analyze` で静的に危険箇所を列挙する
2. `Probe` で環境側の実測結果を取る
3. 両者を突き合わせて、本当に手を入れるべきファイルを絞る

これは「全部読む」「全部直す」を避ける設計です。500+ ファイル規模なら、この発想でないと人間が先に摩耗します。

### 1-2. 制約の捉え方が現実的

制約の切り分けも良いです。

- EDR はファイル破損を起こすレベルの脅威
- 64bit Office は互換性崩れ
- OneDrive / SharePoint はパスと保存挙動の変質
- Shell / PowerShell は実行ポリシー・EDR による実行時ブロック

この4系統を同じ「危険」で平坦に扱わず、発生地点の違いとして扱っているのが賢いです。特に `Analyze` で環境依存を独立カテゴリに引き上げた判断は正しいです。移行レビューでは、PtrSafe 修正より先に「パス前提が死んでいるコード」を見つけたい場面が多いので、優先度の置き方が実務寄りです。

### 1-3. 実測を設計の中心に置いている

このプロジェクトの価値は、推測ではなく実測を中核に置いたことです。

`docs/env-test-results/environment-report.md` と `Probe` 実装を見ると、`ThisWorkbook.Path`、`AutoSave`、`Dir()`、`SaveAs`、`Workbooks.Open` の振る舞いを実際の環境で観測し、その結果を `Analyze` の severity やツールチップへ戻しています。これはかなり良い循環です。

静的解析だけだと「危なそう」で止まりがちですが、ここでは「実際にどこが死ぬか」まで落ちています。この差は大きいです。

## 2. 設計レビュー

### 2-1. ツール分割は妥当

ツールの分割は全体にうまくいっています。

- `Extract`: コード可視化
- `Analyze`: 主役。静的解析、CSV、HTML、サニタイズ
- `Diff`: 修正確認
- `Unlock`: 保護解除
- `Probe`: 環境実測

機能の境界が比較的明快で、役割がかぶっていません。特に `Analyze` を中心に置きつつ、周辺ツールを補助に回している構造は安定しています。

### 2-2. Analyze / Probe の分離は正解

ここはこのプロジェクトの一番良い設計です。

`Analyze` は Excel を開かずに OLE2 + MS-OVBA を読む。`Probe` は逆に Excel COM を使って実環境で挙動を見に行く。この分離が効いています。

- 解析側は安全・高速・一括向き
- 実測側は遅いが現実に近い

この2本を混ぜていないので、責務が壊れていません。特に `Probe` の storage probe を main probe から分けた判断は良いです。`ThisWorkbook.Path` のような現地依存の問題は、一時 xlsm では見えないからです。

### 2-3. 一人運用前提の UX は首尾一貫している

BAT エントリ、ドラッグ＆ドロップ、`pause`、HTML ビューア、timestamp 付き output ディレクトリ。全部、単独運用の実務ツールとして筋が通っています。

自動化基盤の美しさより、「Windows 上でその場で使える」ことを優先している。これは対象に合っています。少々泥臭いですが、現場の湯沸かしとしては正しい設計です。

### 2-4. ただしコアはかなり一枚岩

一方で、`lib/VBAToolkit.psm1` に責務が集まりすぎています。

このファイルには少なくとも、以下が同居しています。

- C# バイト操作実装
- OLE2 パーサ
- VBA 圧縮・展開
- HTML ベース生成
- 出力管理
- ログ
- 分析パターン定義
- 代替 API データベース

現時点では回っていますが、保守の観点では重いです。今後さらに検知パターンや UI を増やすと、変更のたびに別領域へ波及しやすくなります。

つまり設計としては「今は強いが、将来は折れやすい一本柱」です。自己使用ツールとしては許容範囲ですが、長く育てるなら分割候補です。

## 3. 実装レビュー

### 3-1. バイナリ解析の実装はかなり本気

`VBAToolkit.psm1` の OLE2 解析と MS-OVBA 圧縮・展開は、実装の芯として十分強いです。PowerShell だけで頑張らず、C# を `Add-Type` で噛ませているのも正しい判断です。

特に良い点はこうです。

- セクタチェーン読み出しを C# に逃がしている
- mini stream も扱っている
- 圧縮・展開を自前で持っている
- ZIP コンテナ (`xlsm`, `xlam`) と OLE2 (`xls`) の両方に対応している

単なる正規表現ツールではなく、「Excel ファイルをちゃんと解剖している」実装です。この層が自前で成立しているのは強いです。

### 3-2. Analyze は実用品としてよくできている

`Analyze.ps1` はかなり完成度が高いです。

- 設定 GUI と一括解析の両立
- CSV 集計
- テキストレポート
- HTML ビューア
- サニタイズ済みコピー生成
- 出力衝突回避
- 代表証拠や担当区分の自動付与

単に検知するだけでなく、「見て判断する」ための補助が一式入っています。とくに HTML ビューアのサイドバー、ミニマップ、ツールチップ、アウトラインは、読み手の視線移動をかなり減らします。

ここはよく詰めています。

### 3-3. サニタイズ方針は慎重で良い

デフォルトで sanitize=true が `Win32 API (Declare)` だけ、という判断は良いです。`environment-report.md` の実測とも一致しています。

- Declare は存在だけでファイル破損リスク
- Shell などは実行時ブロックであり、ファイルそのものは壊さない

この差をちゃんと扱っているので、過剰な自動改変になっていません。道具として節度があります。

しかも実装は Declare 行だけでなく、宣言された API 名の呼び出し側もコメント対象にできる作りです。これは docs の説明より実装のほうが少し踏み込んでいます。

### 3-4. Probe は「観測装置」として成立している

`Probe.ps1` は良い意味で愚直です。

- xlsm を作る
- VBA を注入する
- 保存する
- 閉じて開き直す
- 実行する
- 結果を記録する

この手順で、EDR の「保存時に壊す」「開き直しで壊す」まで見ているのは良いです。単に `Run` して終わりだと取れない情報です。

さらに storage probe 側では、`MaskPath`、`Path Type`、`Dir()`、`AutoSave`、`Open`、`SaveAs`、外部リンクまで見ています。OneDrive / SharePoint 問題を観測単位へ落とせているので、道具として意味があります。

### 3-5. 周辺ツールも役割が明確

`Extract.ps1` は module collision を避けるため per-file subfolder を作っていて、地味ですが実用的です。`Diff.ps1` は greedy diff で軽く回しつつ、HTML で目視比較を成立させています。`Unlock.ps1` も「原本を触らず output にコピー」を守っていて、安全側です。

このあたりも雑ではありません。

## 4. 実行・運用レビュー

### 4-1. プロジェクトは実測で前進している

git 履歴を見ると、流れが良いです。

- `46e8bf1`: env 拡張の実装
- `d103fb1`: probe 実測に合わせた tooltip 全面更新
- `dc3a81b`: 優先順や RiskLevel などの調整
- `5891c25`: docs を 3 本へ整理し現行実装へ揃える

つまり、仕様を先に神格化していません。実測して、揃えて、整理している。この順番は健全です。

### 4-2. 出力設計も素直

`output/<timestamp>_toolname/` の運用は扱いやすいです。比較しやすく、事故が少なく、前回実行も残ります。

実務ツールは派手さより「昨日の結果が消えない」ことのほうが大事です。ここはちゃんと押さえています。

### 4-3. ただし自動テストはまだ薄い

`test/` には以下があります。

- `test_replacements.ps1`
- `rebuild_tests.ps1`
- `csharp-check/Check.ps1`

方向としては良いです。特に `Add-Type` 可否を独立確認する発想は賢いです。

ただ、全体としてはまだ「回して確認する」寄りで、回帰検知の厚みは強くありません。

不足しているのは例えばこういう層です。

- 既知 fixture に対する `Analyze` 出力件数の固定検証
- `Extract` の module count / line count の期待値確認
- `Diff` の差分件数の再現検証
- `Unlock` の成功 / 不成功条件の機械判定

今は作者の頭で回せる規模ですが、今後パターンを足していくと、気づかないズレが出やすくなります。

## 5. 強い点

### 5-1. 構想と実装の一致度が高い

README、docs、コードの中心思想が揃っています。ここはかなり大きいです。

### 5-2. 実測で静的解析を補強している

`Probe` の結果を `Analyze` の severity やツールチップに戻しているため、レポートが机上の空論になっていません。

### 5-3. 自動化と目視確認の配分が良い

CSV で全体を切り、HTML で個別を読む。両方あるので、人間が必要なところだけ読む運用に向いています。

### 5-4. 危険度の優先付けが実務寄り

環境依存を互換性より前面に出したこと、Declare のみを既定サニタイズ対象にしたこと、この二つは判断が正しいです。

### 5-5. 非破壊を意識している

`Unlock` のコピー運用、sanitized copy の別出力、timestamp 管理など、原本を壊さない設計が徹底されています。

## 6. 主な負債・リスク

### 6-1. `VBAToolkit.psm1` の肥大化

最大の技術負債はこれです。今は便利ですが、変更単位が大きすぎます。分析パターンの更新、HTML 調整、OLE2 修正が同じファイルへ集まっているので、将来の見通しは少々悪いです。

### 6-2. 静的解析はあくまでヒューリスティック

`Analyze` は regex ベースです。移行レビュー用途としては十分ですが、構文木レベルの厳密性はありません。

取りこぼしや過検知の余地がある箇所は、例えばこうです。

- 行継続をまたぐ記述
- 文字列分解された動的構築
- 文脈依存の `Workbooks.Open` / `SaveAs`
- `Path` を経由しない相対処理

つまり「修正候補の選別」には強いが、「意味論的な完全判定」を期待しすぎると危ない、という立ち位置です。

### 6-3. サニタイズは OLE2 ストリーム再配置をしない

`Write-Ole2Stream` は既存チェーン内へ上書きする実装で、容量超過時は例外になります。つまり、コメント追加で圧縮後サイズが増えすぎるケースでは、理屈上は詰まる可能性があります。

現状の用途では大きな問題になっていないはずですが、これは実装上の天井です。長期的には「再配置しない OLE2 書き戻し」の制約が残ります。

### 6-4. `Diff` は近似アルゴリズム

`Diff.ps1` 自身にコメントがある通り、full LCS ではなく greedy approximate diff です。閲覧用としては十分ですが、厳密差分の基盤ではありません。

表示道具としては妥当です。ただし、将来もし diff 結果を機械判定へ使うなら足りません。

### 6-5. `Unlock` は Excel COM と旧形式変換に依存する

`.xlsm/.xlam -> .xls -> DPB パッチ -> 戻す` という流れは現実的ですが、Excel バージョンや信頼設定への依存が濃いです。成功条件が環境寄りなので、堅牢性は他ツールより低めです。

### 6-6. テストの中心がまだ手動寄り

作者が使うには十分ですが、将来的な改修耐性はテストの薄さに引っ張られます。今後 `Analyze` のパターンが増えるほど、この問題は目立つはずです。

## 7. 仕様と実装の小さなズレ

大枠はかなり揃っていますが、細部では少しズレがあります。

### 7-1. ログファイル名

`docs/overview.md` では実行ログが `vba-devkit.log` となっていますが、実装 (`Write-VbaLog`) は `vba-toolkit.log` を書いています。ここは明確にずれています。

### 7-2. `MigrationClass` の説明

`docs/analyze-spec.md` では `MigrationClass` を日本語の区分で説明していますが、実装 (`Build-AnalyzeCsvRow`) が実際に出力する値は英語コードです。

実装側の主な値:

- `NoChange`
- `MinorFix`
- `NeedsReplacement`
- `Rebuild`
- `StorageReview`
- `RefactorPathHandling`
- `ReviewSaveFlow`
- `ReviewExternalLink`

ここは仕様書が古いというより、抽象化された説明のまま止まっている状態です。CSV を読む側には実装値のほうが事実です。

### 7-3. storage probe の出力位置

overview から読むと `probe` の output は `output/` 配下で揃って見えますが、storage probe のエクスポート実装は workbook 側の `probe-export` フォルダです。自分用なら大勢に影響しませんが、読み手が増えるなら一言あると混乱しにくいです。

## 8. 結論

このプロジェクトは成功しています。

少なくとも、構想倒れではありません。対象の制約に対して、道具の分け方が合っており、実装も十分追いついています。とくに `Analyze` と `Probe` の二本立ては強く、実測を仕様へ戻している点はかなり良いです。

一方で、今後の論点は「作れるか」ではなく「どこまで育てるか」です。現状は一人運用の実務ツールとしてかなり完成していますが、保守性は `VBAToolkit.psm1` の一極集中とテストの薄さに少しずつ支配されます。

なので、現時点の評価を一文で言うならこうです。

「狙いは当たり、実装も届いている。次の問題は機能不足ではなく、肥大化と検証密度である。」

今の段階は、もう試作ではありません。十分に使うための道具です。
